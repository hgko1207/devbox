## Problem Statement

devbox는 개발자가 자주 쓰는 웹 도구를 한 사이트에 모은 프로젝트다. 이미
it-tools, DevToys, bestdev.tools 등 **폭넓은 영문 도구 모음**이 다수 존재한다
(bestdev.tools만 해도 100개+ / 17개 카테고리). 목표는 **포트폴리오·공개용 사이드
프로젝트**이며, 잘 되면 제품화 가능성도 열어둔다. 따라서 핵심 질문은 두 가지다:
(1) 수십 개의 기존 모음과 어떻게 차별화하는가, (2) 도구를 늘려갈 때 "산만한
아카이브"가 되지 않게 어떻게 정체성을 지키는가.

## What Makes This Cool

- **한국어 네이티브 경험** — 한국 개발자가 "JSON 정렬"을 검색하면 영문 도구로
  가는 게 현실이다. UI·에러 메시지·예제·빈 상태까지 자연스러운 한국어로 채우는
  것은 기존 강자들(전부 영문 우선)이 비워둔 칸이다.
- **검증된 정확함** — 단순한 도구도 엣지케이스까지 안정적으로. 이미 JSON 도구가
  수 MB 처리 + 정밀한 에러 위치(줄·열·이유) + Web Worker 비차단으로 이를 증명했다.
- **경험 레이어** — ⌘K 명령 팔레트·키보드 우선·오프라인(PWA)·일관된 디자인 시스템.
  개별 도구의 합보다 이 "쓰는 맛"이 "잘 만든 사람"을 더 강하게 증명한다.

## Constraints

- 100% 클라이언트 처리, 백엔드 없음, GitHub Pages 정적 배포.
- 추적·광고·회원가입 없음, 외부 요청 0(시스템 폰트 사용).
- 스택 고정: React + TS + Vite + Tailwind. 도구 레지스트리 기반 자동 라우팅/홈 구조 유지.
- 솔로 개발 · 시간 제약 → "폭으로 100개"는 비현실적이며 전략적으로도 비추천.

## Premises

- **P1 (수정됨):** 도구 개수를 인위적으로 제한하지 않는다. 핵심은 "소수"가 아니라
  **품질 게이트** — 추가되는 모든 도구는 *한국어 네이티브 + 정확함 + 적정 깊이*
  바를 통과해야 한다. 많아져도 "산만한 아카이브"가 아니라 "전부 잘 만든 N개"가 된다.
  ("엄선"은 개수가 아니라 기준이다. **제목의 '큐레이션'도 선별적 포함이 아니라 이
  품질 게이트를 가리킨다.**)
- **P2:** 차별화 축 셋 = ①한국어 네이티브 ②정확함/안정성(단순 도구도 엣지케이스까지)
  ③깊이는 필요한 도구에만(JSON류처럼 기능이 더 들어가야 하는 것).
- **P3:** 개별 도구보다 '경험 레이어'(⌘K·키보드·PWA·일관 디자인)가 포트폴리오 임팩트가 크다.
- **P4:** '깊이'는 30초 보는 채용 담당자에게 저절로 안 보인다 → README 사례·벤치마크·
  도구 내 안내로 '읽히게' 만들어야 한다.

## Approaches Considered

### Approach A: 폭을 빠르게 — 빈출 도구로 '진짜 모음' 만들기  [채택 — 1단계]
- 요약: 자주 쓰는 단순 도구 4~5개를 '같은 품질 바'로 빠르게 추가. JSON 패턴·레지스트리 재사용.
- Effort: human ~1주 / CC ~1–2시간 · Risk: Low · Completeness 7/10
- Pros: 단기간 '쓸 만한 컬렉션' 완성 · ⌘K 팔레트가 의미 생기는 최소 도구 수 확보.
- Cons: 경험 레이어·깊이 가시화는 뒤로 밀림.
- Reuses: registry, JSON 도구 구조, Web Worker 패턴.

### Approach B: 경험 레이어 먼저 — 플랫폼처럼  [이상적 아키텍처 — 2단계]
- 요약: ⌘K 명령 팔레트 · 키보드 우선 · PWA/오프라인 · 공유 디자인 시스템 + 도구 '셸'
  (입력/출력 패널·복사·상태바 공통화)을 먼저. 이후 모든 도구가 싸고 일관됨.
- Effort: human ~1–2주 / CC ~2–4시간 · Risk: Med · Completeness 8/10
- Pros: 폴리시 자체가 차별점(P3) · 30초에 보이는 wow(P4) · 향후 도구 단가 급감 · '플랫폼 사고' 신호.
- Cons: 도구가 1개뿐인 지금은 시기상조 → 도구 몇 개가 먼저 있어야 함.
- Reuses: 기존 Layout / registry / theme.

### Approach C: 깊이 간판 테마 — 데이터 포맷 쐐기  [창의적/측면 돌파 — 3단계]
- 요약: 한 테마(데이터 포맷)에 올인 — JSON 패밀리를 한국어권 최고로:
  JSONPath 쿼리 · JSON↔YAML/CSV · TS 타입 생성 · (기존)트리/diff. README 사례 + 벤치마크로 깊이 가시화.
- Effort: human ~1주 / CC ~1–2시간 · Risk: Low · Completeness 7/10
- Pros: 부정할 수 없는 '깊이' 간판 1개(포트폴리오에 읽힘) · 기존 JSON 자산 전부 재사용.
- Cons: 컬렉션 정체성 약함 · 'JSON만' 인상 위험.
- Reuses: JSON 도구 + Worker 전체.

## Recommended Approach

**A → B → C 순서. 지금 당장은 A.** B(경험 레이어)와 C(간판)는 모두 도구가 몇 개는
있어야 의미가 생긴다(⌘K 팔레트는 항목이 있어야, "컬렉션"은 컬렉션이 있어야 빛난다).
A로 토대(도구 4~5개 + 품질 바 확립)를 깔면 B가 자연스럽게 싸지고, C는 JSON에서 계속
굴린다. 셋을 순서대로 완성하는 것이 완성도 9/10 경로다.

구체화:
- **1단계 (A):** 빈출 단순 도구 추가 — 후보: Base64 인코드/디코드, JWT **디코더(클레임
  표시 전용, 서명 검증은 하지 않음)**, URL 인코드/디코드, 타임스탬프(Unix↔날짜) 변환,
  UUID 생성. 전부 즉시성·정확성 중심, 한국어.
- **2단계 (B):** 공통 도구 셸 컴포넌트화 → **⌘K/Ctrl+K** 명령 팔레트(레지스트리 기반
  자동 등록, **MVP는 '도구로 이동'만**; 도구 내 액션·퍼지검색은 이후) → 키보드 단축키 →
  디자인 토큰 정리. **PWA/오프라인은 별도 게이트**(Open Questions 참조)로, B에 자동 포함하지 않는다.
- **3단계 (C):** JSON 패밀리 깊이 간판 — JSONPath, JSON↔YAML/CSV, JSON→TS 타입 생성 +
  README 벤치마크/사례화.

(완성도 9/10은 개별 점수 합이 아니라 **세 단계가 서로의 전제를 채워주기 때문**이다:
A가 B의 "팔레트에 담을 항목"과 C의 "간판이 묻히지 않을 맥락"을 만든다. 단독 실행 시 각
7~8/10이지만 순차 완성 시 빈틈이 메워진다.)

## Open Questions

- **분류 체계 (결정됨):** A단계는 **단일 평면 목록**으로 진행한다 — A-2/A-3의 도구 추가는
  카테고리 결정에 의존하지 않는다(레지스트리에 등록만 하면 홈에 노출). 테마/카테고리 그룹핑은
  도구가 6~8개 모이는 **B단계의 홈 화면 작업**으로 미룬다.
- **PWA/오프라인 (열린 게이트):** B의 일부로 자동 포함하지 않는다. 비용(서비스워커 캐싱,
  manifest, base-path 스코프, 업데이트 전략)이 가장 크고 포트폴리오 페이오프는 가장 덜
  분명하므로, B의 다른 항목(셸·⌘K)을 끝낸 뒤 **별도로 할지 결정**한다.
- 한국어 SEO/sitemap/OG: **"나중에 / 제품화 시"** 버킷. 현재 A/B/C 계획에는 넣지 않는다(YAGNI).

## Success Criteria (= 품질 게이트 = devbox의 정의)

모든 도구는 추가 전 다음을 통과해야 한다(이 체크리스트가 곧 차별화의 실체):
1. **한국어:** UI·에러 메시지·예제·빈 상태가 자연스러운 한국어.
2. **정확성:** 대표 엣지케이스 통과 — 빈 입력 / **거대 입력(목표: ~10MB까지 무지연, 초과 시
   경고 후 처리)** / 잘못된 입력의 명확한 에러 / 유니코드·한글.
3. **즉시성:** **작은 입력은 메인 스레드 <100ms, 큰 입력은 Web Worker로 오프로드(+진행 표시).**
4. **일관성:** 공통 셸/디자인 토큰 사용. **단, 이 항목은 2단계(B) 이후의 도구에만 강제**한다.
   1단계(A)에서 추가한 도구는 B에서 셸로 리트로핏한다(게이트 통과 시점 예외).
5. **프라이버시:** 네트워크 요청 0. (PWA 도입 시에도 외부 fetch 금지 — 빌드 산출물에 폰트/CDN 유입 없는지 확인.)
6. **접근성:** 키보드만으로 핵심 동작 가능 + 다크모드. **복사 실패 대비**(클립보드 권한 거부·비보안
   컨텍스트) 폴백 — 실패 시 안내 메시지 + 수동 선택 가능.

## 확정된 방향 (premise gate)
- **타겟 = 주로 국내(한국)** → 한국어 네이티브는 강점(확정). README 영문 병기는 보너스.
- **시퀀스 = 테마 깊이 우선(절충)**: 순수 C-first(1도구=방치)도, 순수 폭 우선(간판 희석)도 아님.
  1. **지금:** 배포(A-0) + JSON 테스트 추가(="정확함" 증명) + 한국어 README(깊이·프라이버시 셀).
  2. **테마 컬렉션:** JSON과 한 가족인 데이터/인코딩 도구(Base64·URL·JWT디코드·타임스탬프)를 품질 게이트로.
  3. **간판 깊이 1방:** JSON↔YAML/CSV 또는 JSONPath.
  4. **경험 레이어(B):** 도구 5개+ 후 ⌘K·셸. PWA 게이트 유지.
